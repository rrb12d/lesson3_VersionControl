Ryan Biggane

Highlight #1 Version Control Basics

Version Control is a system that records the changes to a file over multiple iterations. This allows the user to return to previous version of the same file at any time. Reverting back to previous versions of a project has many advantages. If you are working in a team setting, you may wish to use version control to keep your individual work protected and easily recognizable. The entire project can be reverted to a previous version if the team makes a mistake. Also, learning version control is a key skill set in the professional world of software/web development. Being able to work on a team using version control is an efficient way to compile work.

Highlight #2 Types of Version Control

Local Version Control is the most common and simple approach taken. Despite this approach being straight forward and intuitive, it's the most prone to error. Data redundancy / file duplication was an epidemic in local version control. To help eliminate this issue, operating systems were equipped with RCS, which keeps patch sets in a special format in storage. RCS can the rebuild any files contents by adding up the patches. Centralized Version Control was developed to allow team work on a project. Putting the work in a remote repo allows multiple people to pull the files, work on them, and push them back with revisions. This is considered to be the standard of version control. This method allows transparency of work between team members and a lot of conrol for the repo administrators. The disadvantages are rather obvious with a centralized system. If the server becomes unavailable for any reason, the entire project comes to a halt for that period of time. Also, if the entire version collection is in one location, the data is especially vulnerable.

Highlight #3 DVCS

To deal with the common issues associated with Local and Centralized version control, the distributed model was developed. By distributing the repo to each client, the data is never holed up in one place. The client recieves the entire project instead of just seeing the latest version of a specific file. There is much less risk involved with server collapse and restoration. Another advantage of using this method is that clients can have several full repos at their disposal. This allows someone to contribute work within different circles of the same project. Essentially, anyone can work on anything in the project without getting in the way. This method is preferable to the hierarchical style of work.

Highlight #4 Git basics

Git was built from the ashes of an old project. The aim of this new system was to emphasize simplicity, strong support, speed, full distribution, and its ability to work with huge projects. To this day, it remains a leading system for non-linear development. Most version control systems keep their records as a list of file changes. They build the version from the ground up, using the different changes to patch up the requested version on the spot. Instead of using this approach, Git simply takes snapshots of the file whenever changes are saved. Git uses a stream of snapshots instead of the list of file changes. Git is mostly local driven, which gives in an unnaturally fast speed. Since everything is in one place, Git doesn't need to communicate with a server for the history of a project.

Highlight #5 Git characteristics

Git uses specific identification for everything that is stored. This ID is called a checksum, which is a 40 character string of hexadecimal characters that are derived from the contents of a file. This method of ID is called SHA-1 hash. Git has three states a file can reside in. The first is committed, which means the data is stored in your local repo. The second is Modified, which means that the file has been changed but not committed to the local repo. The last state is staged, which means the the modified file is marked for the next commit snapshot.
